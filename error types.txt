Types of logical errors:
    1. Disjoint properties between the same pair of entities
        1.1. Disjointness between two properties -- Fails, requires at least sh:disjoint to be fixed
        1.2. Disjointness between a property and a property chain -- Fails, can be fixed with custom rules like (1)
    2. Disjoint classes assigned to the same entity -- Fails, can be fixed with (4)
    3. Violation of domain or range constraints of a relation -- Passes, but class hierarchy should be minded (challenge 1.1)
    4. Irreflexivity Violation: A property is owl:IrreflexiveProperty, but the data contains a triple $(A, P, A)$ -- Fails, requires custom rules
    5. Asymmetry Violation: $(A, P, B)$ and $(B, P, A)$ both exist for an owl:AsymmetricProperty -- Fails, requires custom rules to handle
    6. Max Cardinality Violation: A node has 3 values for a property restricted to maxQualifiedCardinality 2 -- Passes
    7. Functional Property Violation: A FunctionalProperty (like hasBirthMother) points to two distinct individuals who are not the same -- Fails, requires custom rules

Common challenges:
    1. Class hierarchy: 
        1.1. If sh:range :Person, then :Man (being subclass of :Person) still won't be passed by SHACL (can be fixed with custom rules like (2))
        1.2. Constraints should be inherited
    2. Property shapes propagation: :Man should get all prop shapes from :Person as well as from parallel shapes like :Ancestor (if not disjoint)
    3. For some constraints (like sh:disjoint) for each original prop all the children and equivalents should be listed as well
    4. Inverse properties should be treated carefully:
        4.1. domain and range constraints -- for inverse properties they need to be inferred, but whatever tool I tried, none gave that result. Protege shows correct inferrals, but when I export them, though with all the checkboxes marked, still there are no inferred domain/range in the ontology.
        4.2. propertyDisjointWith -- not one reasoner inferred them; probably need to be specified in the ontology? but family ontology doesn't have it
        4.3. propertyChainAxiom -- inverse triples should be inferred dynamically (3), otherwise if a property chain is stated in the inverses, it won't be recognized.

(1)
:UncleInferenceShape
    a sh:NodeShape ;
    sh:targetSubjectsOf :isBrotherOf ;
    sh:rule [
        a sh:TripleRule ;
        sh:subject sh:this ;
        sh:predicate :isUncleOf ;
        sh:object [
            sh:path ( :isBrotherOf :hasChild )
        ] ;
    ] .

(3)
:DynamicInverseShape
    a sh:NodeShape ;
    sh:targetClass rdfs:Resource ;
    sh:rule [
        a sh:SPARQLRule ;
        sh:construct """
            CONSTRUCT {
                ?o ?inv $this .
            }
            WHERE {
                # 1. Find any property ?p connected to the current node
                $this ?p ?o .
                
                # 2. Find the inverse property definition in the metadata
                # We check both directions because owl:inverseOf is symmetric
                { ?p owl:inverseOf ?inv . }
                UNION
                { ?inv owl:inverseOf ?p . }
                
                # 3. Optimization: only construct if ?o is an IRI (to avoid literal subjects)
                FILTER (isIRI(?o))
            }
        """ ;
    ] .

(4)
:ManDisjointnessShape
    a sh:NodeShape ;
    sh:targetClass :Man ;
    # Violation if the node is also a Woman
    sh:not [
        sh:class :Woman ;
    ] ;
    sh:message "A node cannot be both a Man and a Woman." .
